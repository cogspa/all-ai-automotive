# ALL AI AUTOMOTIVE v2

A YMM-aware diagnostic and installation knowledge base for automotive parts.
Built on a structured bulletin schema with NHTSA integration and per-part
content templates, so every page reads like it was written for that specific
part — not regenerated from the same boilerplate.

## What changed in v2

The legacy site had three core problems:

1. **35 of 39 bulletins read identically.** The generator fell through to a
   generic "diagnostic advice document" template for everything that wasn't
   one of 4 starter bulletins.
2. **No vehicle awareness.** A user looking for "2010 Honda Accord brake
   pads" had to scroll a flat list of every bulletin in the index.
3. **Substring search only.** Typos and synonyms missed.

v2 addresses each:

| Problem | Fix |
|---------|-----|
| Identical pages | New schema with structured `symptoms`, `steps`, `torqueSpecs`, `causes`, etc. — generator only renders sections the bulletin actually has. |
| Per-part content | `data/part-content.cjs` — a library of differentiated content keyed by part ID. Currently covers starter, alternator, brake-pad, brake-caliper, ignition-coil, oxygen-sensor, wheel-hub, and a quality fallback. Add more by editing this one file. |
| No YMM | Year/Make/Model cascade backed by NHTSA vPIC API (free, no key). Bulletins filter by applicability automatically. |
| Search | Fuse.js fuzzy search weighted across `symptoms`, `partType`, `title`, `applicability`, `causes`, `summary`. |
| No recall data | NHTSA Recalls API integration — once a vehicle is selected, open recalls render in their own panel. |
| One bulletin type | `diagnostic` and `installation` are now separate types with different page layouts. |

## Quick start

```bash
npm install
npm run rebuild-content    # migrate legacy index + regenerate HTML pages
npm run dev                # Vite dev server
```

The legacy `searchIndex.json` is preserved at `public/searchIndex.legacy.json`
on first migration. Run `npm run migrate` again any time the source legacy
data updates.

## Project structure

```
data/
  schema.md             # Bulletin v2 schema docs
  parts-taxonomy.json   # 8 categories, ~50 part types
  part-content.cjs      # Differentiated content per part
public/
  index.html            # YMM selector + search + facets
  main.js               # Fuse.js search + vPIC + recall integration
  style.css
  searchIndex.json      # v2 bulletin data (generated by migrate)
  searchIndex.legacy.json  # Backup of pre-migration data
  data/parts-taxonomy.json # Served copy, for the frontend
  html_bulletins/*.html # Generated per-bulletin pages
scripts/
  migrate.cjs           # Legacy → v2 migration
  generateHtmlBulletins.cjs # v2 → HTML pages
```

## Data sources

All public, free, no API key required:

- **NHTSA vPIC API** — Year/Make/Model cascade
  `https://vpic.nhtsa.dot.gov/api/`
- **NHTSA Recalls API** — Open recalls by YMM
  `https://api.nhtsa.gov/recalls/recallsByVehicle`

Note: NHTSA APIs do not require authentication but they do enforce reasonable
rate limits. If you're hammering them in development, consider caching
responses locally.

## Schema overview

Every bulletin in `searchIndex.json` looks roughly like this. Required fields
vary by `type`:

```js
{
  id: "TBS0131",
  type: "installation",          // or "diagnostic", "tsb", "recall"
  title: "Starter Installation Guide #131",
  partType: "starter",
  partCategory: "engine-electrical",
  difficulty: "intermediate",
  estTimeMinutes: 45,
  applicability: [               // empty array = universal
    { make: "Nissan", yearStart: 2007, yearEnd: 2012 }
  ],
  summary: "...",
  tools: [{ name, required, note }],
  parts: [{ name, quantity, note }],
  steps: [{ title, instruction, spec, warning }],
  torqueSpecs: [{ fastener, value, unit, sequence }],
  warnings: [...],
  tips: [...],
  relatedBulletins: [...]
}
```

Full field-by-field rules in `data/schema.md`.

## Adding new part content

1. Open `data/part-content.cjs`.
2. Add a top-level entry keyed by the part ID (matching `parts-taxonomy.json`).
3. Provide both `diagnostic` and `installation` blocks — even if one is sparse.
4. Run `npm run rebuild-content`.

The generator detects which sections are populated and only renders those, so
a sparse part still produces a clean page.

## Adding new bulletins manually

The migration script handles the legacy import. For new bulletins, you can
either append to the legacy file before migrating, or edit `searchIndex.json`
directly post-migration and just run `npm run generate`.

## Browser support

Modern evergreen browsers. The frontend uses ES modules and dynamic imports
(Fuse.js from jsdelivr CDN), so IE/old Edge will not work.
